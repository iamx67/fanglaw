# Fanglaw Status

Статус на текущий момент: фундамент проекта собран, основной auth/character flow работает, Postgres runtime переведён на новую split-модель данных.

## Коротко простыми словами

Если совсем без технического жаргона, то сейчас проект работает так:

- человек создаёт аккаунт
- отдельно создаёт персонажа
- отдельно выбирает внешность
- потом заходит в игру
- игра и сайт начинают говорить об одном и том же персонаже

Раньше часть этого жила в старой “куче” данных.
Сейчас основная модель уже разрезана на понятные части.

## Что уже сделано

### Архитектура и backend

- есть backend на `Node.js + TypeScript + Colyseus + Express`
- есть сервисный слой поверх persistence
- есть миграции PostgreSQL
- есть split-таблицы персонажа
- есть backfill из legacy

### Auth и персонажи

- регистрация аккаунта отделена от создания персонажа
- логин создаёт только auth-сессию
- персонаж создаётся отдельным API
- внешность сохраняется отдельным API

### Сайт

- `/` — auth
- `/create-character.html` — create character
- `/appearance.html` — appearance
- в шапке снова обычная кнопка `Играть`
- локальный сайт раздаёт и страницы сайта, и `catlaw.html`

### Игра

- вход в игру идёт по `sessionToken`
- визуал и формат данных клиента не переписывались
- сервер-авторитетный мир уже работает
- текущий клиент продолжает читать `appearanceJson` и `skillsJson`

## Что было обновлено совсем недавно

### 1. Игра переведена на новую модель данных в Postgres runtime

Сделано:

- backend теперь собирает персонажа из:
  - `auth_characters`
  - `character_profiles`
  - `character_appearances`
  - `character_progression`

Это значит:

- профиль персонажа
- внешность
- прогрессия

теперь живут не в одной legacy-таблице, а в правильных отдельных таблицах.

### 2. Legacy не сломан

При этом не было сделано опасного жёсткого отрезания.

`player_profiles` пока оставлен:

- для world snapshot
- для совместимости
- как fallback на переходный период

### 3. Миграция JSON -> Postgres тоже обновлена

Теперь перенос старых данных не теряет:

- `tribe`
- `gender`
- `appearance`
- `skills`

и записывает их не только в legacy-таблицу, но и в новые `character_*` таблицы.

## Что ещё осталось

- полный cleanup legacy `player_profiles`
- cleanup legacy-имени `playerId`
- возможно, отдельный split-store для file fallback
- дальнейшее развитие самой игры поверх уже готового фундамента

## Что это означает для разработки

Главный смысл такой:

- архитектурный этап в основном пройден
- можно меньше чинить фундамент
- можно больше заниматься самой игрой
