## 1. Что это за проект

`Fanglaw` / `Catlaw` — браузерная MMO-игра про котов.

Проект сейчас находится на стадии **рабочего сетевого прототипа**, а не завершённой игры.

Уже сделано:

- базовый Colyseus-сервер;
- один общий мир;
- подключение нескольких игроков;
- сервер-авторитетное движение;
- guest identity через `playerId`;
- простой reconnect;
- простое файловое сохранение игрока;
- деплой на VPS;
- раздача web-клиента через домен.

Ещё не сделано:

- полноценная игровая механика;
- территории, NPC, квесты, инвентарь;
- нормальная БД;
- HTTPS/wss доведённый до конца для браузерного прод-режима;
- production-auth;
- полноценная админка;
- автодеплой;
- нормальный внешний SSH-доступ для комфортного деплоя.

Важно: инфраструктура уже **достаточно стабильна**, чтобы начинать разработку самой игры.

- `Node.js 22`

- `@colyseus/ws-transport`
- `@colyseus/schema`
- `Express`
- `PM2`

### 2.2. Клиент

Целевая идея проекта:

- `Godot 4.6.1 .NET`
- в будущем возможен перенос части логики на `C#`

- рабочий клиентский прототип сделан **на GDScript**
- используется плагин `godot_colyseus`

Причина: это был самый короткий путь к стабильной клиент-серверной связке.

### 2.3. Веб-версия

Сейчас игра может быть собрана как **Godot Web export**.

Важно:

- текущий web-клиент основан на GDScript-сцене;
- Godot 4 `.NET/C#` web-export не является текущей рабочей целью;
- для браузерного запуска нужен `HTTPS`, потому что Godot Web требует `Secure Context`.

## 5. Текущее сетевое устройство проекта

### 5.1. Общая модель мира

Сервер больше не работает как временная тестовая комната, которая создаётся только когда приходит игрок.

Текущая модель:

- сервер стартует;
- поднимает один общий мир `cats`;
- этот мир живёт постоянно, пока жив процесс сервера;
- игроки заходят в один и тот же экземпляр мира;
- мир не уничтожается при `0` игроках.

Это уже ближе к настоящему MMO-подходу, чем обычная временная “комната”.

### 5.2. Идентичность игрока

Идентичность игрока строится не на `sessionId`, а на `playerId`.

Сейчас это guest identity:

- у клиента есть локально сохранённый `playerId`;
- сервер принимает `playerId` при входе;
- состояние игрока хранится по `playerId`;
- `sessionId` используется только как идентификатор текущего сетевого соединения.

Это важно для:

- reconnect;
- сохранения позиции;
- будущего инвентаря;
- прогресса;
- дальнейшего перехода к аккаунтам.

### 5.3. Движение

Движение уже переведено на сервер-авторитетную модель.

Это значит:

- клиент не отправляет готовые `x/y`;
- клиент отправляет input-направление;
- сервер сам считает позицию;
- сервер ограничивает скорость;
- сервер ограничивает координаты границами мира;
- клиент только рисует и сглаживает.

Простая подмена позиции на клиенте больше не должна телепортировать персонажа.

### 5.4. Reconnect

Reconnect уже есть в прототипном виде.

Модель сейчас такая:

- при разрыве сервер держит слот игрока около `20 секунд`;
- клиент может попытаться вернуться в ту же сессию;
- если успевает — возвращается тот же кот;
- если не успевает — старое соединение считается протухшим, и вход уже считается новым.

Это ещё не production-auth reconnect, но для прототипа этого достаточно.

### 5.5. Сохранение данных

Сейчас используется простое файловое сохранение.

Файл:

- `server/data/players.json`

Там хранятся guest-профили игроков:

- `playerId`
- `name`
- `x`
- `y`
- временные поля профиля

Это не полноценная БД, но для текущего этапа достаточно.

## 6. Архитектура сервера

### 6.1. Точка входа

Файл:

- `server/src/index.ts`

Что делает:

- создаёт `Express`
- создаёт `Colyseus Server`
- использует `WebSocketTransport`
- читает:
  - `PORT`
  - `PUBLIC_URL`
  - `WORLD_ROOM_NAME`
  - `WORLD_KEY`
- регистрирует комнату `cats`
- загружает player store
- запускает сервер
- сразу создаёт persistent world room через `matchMaker.createRoom(...)`

HTTP root `/` отвечает JSON-статусом сервера.

### 6.2. Комната мира

Файл:

- `server/src/rooms/CatRoom.ts`

Что важно:

- `autoDispose = false`
- `maxClients = 100`
- мир хранит `state.players`
- ключ игроков: `playerId`
- обрабатывает:
  - `ping`
  - `move`
- делает simulation tick
- даёт reconnect window
- сохраняет игрока при уходе или таймауте reconnect

### 6.3. Схема состояния

Файл:

- `server/src/rooms/schema/WorldState.ts`

Текущий `Player` содержит:

- `playerId`
- `sessionId`
- `name`
- `x`
- `y`
- `connected`

### 6.4. Persistence

Файл:

- `server/src/persistence/PlayerIdentityStore.ts`

Что делает:

- грузит `players.json`
- создаёт guest profile при первом входе
- сохраняет snapshot игрока
- делает debounced flush на диск

## 7. Архитектура клиента

### 7.1. Главная сцена

Файлы:

- `client/main.tscn`
- `client/scripts/main.gd`
- `client/scripts/cat_avatar.gd`

Клиент сейчас умеет:

- вводить имя;
- подключаться в мир `cats`;
- хранить `playerId`;
- хранить `reconnection_token`;
- показывать состояние подключения;
- отправлять `move`;
- отправлять `ping`;
- рисовать локального и удалённых игроков;
- сглаживать движение;
- восстанавливаться после reconnect.

### 7.2. Конфиг endpoint

Файл:

- `client/server_config.cfg`

Назначение:

- desktop-клиент по умолчанию подключается к указанному endpoint

Текущее значение в репозитории:

- `ws://fanglaw1.ru`

### 7.3. Web-режим клиента

В `client/scripts/main.gd` добавлена логика:

- если игра запущена в браузере;
- endpoint берётся автоматически из `window.location`

То есть в web-режиме клиент старается подключаться к тому же домену, откуда открыт сайт.

Это нужно для single-domain схемы:

- сайт и websocket/backend живут на одном домене

### 7.4. Плагин Colyseus

Файлы:

- `client/addons/godot_colyseus/lib/client.gd`
- `client/addons/godot_colyseus/lib/room.gd`

Плагин локально патчен под текущий Colyseus:

- понимает новый matchmaking response;
- правильно добавляет `reconnectionToken` в websocket reconnect URL.

Эти изменения критичны. Не обновлять плагин без проверки.

## 8. Веб-версия и домен

### 8.1. Web export

Готовая web-сборка лежит в:

- `client/web_export`

Сейчас там лежат, как минимум:

- `catlaw.html`
- `catlaw.js`
- `catlaw.wasm`
- `catlaw.pck`

Важно:

- стартовый файл называется **`catlaw.html`**, а не `index.html`
- это влияет на конфиг `nginx`

### 8.2. Single-domain схема

Пока нет отдельного `api.` поддомена, используется один домен:

- `fanglaw1.ru`

Через него одновременно идут:

- статические web-файлы игры
- `POST /matchmake/...`
- websocket пути Colyseus

### 8.3. Nginx

Подготовленный конфиг:

- `server/deploy/nginx/fanglaw1.ru.single-domain.http.conf`

Он рассчитан на:

- `root /var/www/fanglaw-web/current`
- `index catlaw.html`
- `location /matchmake/` -> Colyseus backend
- `location ~ ^/[A-Za-z0-9_-]+/[A-Za-z0-9_-]+$` -> websocket Colyseus
- `location /` -> web-клиент

Также в конфиге уже предусмотрены заголовки для Godot Web:

- `Cross-Origin-Opener-Policy`
- `Cross-Origin-Embedder-Policy`
- `Cross-Origin-Resource-Policy`

### 8.4. Текущий статус доменной версии

На момент обновления документа:

- `http://fanglaw1.ru/` уже должен отдавать `catlaw.html`
- `http://fanglaw1.ru/api-health` уже должен отдавать JSON сервера
- браузерная игра всё ещё требует `HTTPS` для полноценного запуска

### 8.5. Что нужно для завершения браузерного запуска

Нужно:

- выпустить TLS-сертификат через Certbot
- перевести сайт на `https://fanglaw1.ru`
- обновить `PUBLIC_URL` сервера на `https://fanglaw1.ru`
- затем desktop endpoint перевести на `wss://fanglaw1.ru`

## 12. Какие решения уже приняты

### 12.1. Модульность и версии

Принято решение держать сервер в:

- `ESM`
- `NodeNext`
- `Node 22`
- совместимой конфигурации Colyseus `0.17.x`

Причина:

- ранее проект ломался на смешении `ESM/CommonJS`;
- была серия ошибок `ERR_REQUIRE_ESM` и конфликтов зависимостей;
- текущая конфигурация уже доведена до рабочего состояния.

### 12.2. Клиент временно на GDScript

Принято решение:

- не переписывать сразу клиент на `C#`
- сначала довести GDScript-прототип до сетевой устойчивости

Причина:

- это сильно ускорило реальный прогресс;
- позволило уже сейчас дойти до VPS и домена.

### 12.3. Один общий мир

Принято решение:

- использовать один persistent world room `cats`

Причина:

- это лучше соответствует цели “один общий MMO-мир”;
- для раннего релиза и нагрузки порядка `~100 online` это нормальная стартовая модель.

### 12.4. Server-authoritative movement

Принято решение:

- сервер считает движение сам

Причина:

- это база под античит;
- без этого любая MMO будет слишком хрупкой.

### 12.5. Простое файловое сохранение

Принято решение:

- не тащить платную БД на раннем этапе

Причина:

- пока хватает JSON persistence;
- позже можно перейти на SQLite или PostgreSQL.
