## 1. Что это за проект

`Catlaw` — браузерная MMO-игра про котов.

Текущая практическая цель проекта:

- иметь минимально рабочую сетевую основу;
- запускать локально сервер и клиент без конфликтов зависимостей;
- подключать нескольких игроков в одну комнату;
- видеть себя и других игроков;
- двигаться по сцене и синхронизировать позиции через сервер.

На данном этапе это не полноценная игра, а рабочий прототип клиент-серверной связки, на котором можно безопасно наращивать механику, интерфейс и инфраструктуру.

## 2. Используемые технологии

### Целевая технологическая связка

- клиент: `Godot 4.6.1 .NET` (долгосрочно предполагается `C#`);
- сервер: `Node.js 22 + TypeScript`;
- сетевой фреймворк: `Colyseus`;
- синхронизация состояния: `@colyseus/schema`.

### Что используется фактически прямо сейчас

Важно: несмотря на целевую ставку на `Godot .NET / C#`, текущий рабочий клиентский прототип сделан **на GDScript**, потому что в проекте уже установлен и используется GDScript-плагин `godot_colyseus`.

То есть на текущий момент:

- сервер реально работает на `TypeScript`;
- клиент реально работает на `GDScript`;
- `C#`-клиент как отдельная реализация **ещё не написан**;
- текущая задача была решена самым коротким путём: сначала добиться стабильной сетевой связки.

Это нормальное инженерное решение для прототипа. Перенос логики клиента на `C#` можно сделать позже, когда базовый сетевой сценарий и структура игры стабилизируются.

## 4. Текущее состояние проекта

### 4.1. Что уже реализовано

Сервер:

- поднимается локально через `npm.cmd run dev` в папке `server`;
- отвечает на `http://localhost:2567/`;
- регистрирует одну комнату `cats`;
- принимает подключения игроков;
- хранит состояние комнаты в `state.players`;
- принимает сообщения `move` и `ping`;
- отвечает сообщением `pong`.

Клиент:

- открывается в Godot;
- показывает простую сцену с вводом имени и кнопкой входа;
- подключается к комнате `cats`;
- отправляет имя при входе;
- получает состояние комнаты;
- отображает локального и удалённых игроков;
- отправляет `move` при движении стрелками;
- показывает базовый статус подключения.

### 4.2. Что это значит на практике

Сейчас уже работает минимальный игровой цикл:

1. запустить сервер;
2. открыть клиент;
3. ввести имя;
4. войти в комнату `cats`;
5. увидеть своего кота;
6. двигаться стрелками;
7. открыть второй клиент и увидеть синхронизацию другого игрока.

Это и есть рабочий минимальный MMO-прототип на уровне сети.

### 5.3. Клиент-серверное взаимодействие

Текущий сетевой сценарий:

1. клиент делает matchmaking-запрос `joinOrCreate` для комнаты `cats`;
2. сервер создаёт или находит комнату;
3. сервер добавляет игрока в `state.players`;
4. клиент получает состояние комнаты;
5. клиент визуализирует всех игроков из `state.players`;
6. клиент при движении отправляет:

```json
{ "x": number, "y": number }
```

7. сервер обновляет координаты игрока;
8. изменения состояния приходят всем клиентам через `Colyseus Schema`;
9. клиент обновляет позиции котов на экране.

Дополнительно:

- клиент шлёт `ping`;
- сервер отвечает `pong` с `sessionId` и `serverTime`.

## 7. Принятые решения

### 7.1. Единый режим модулей для сервера

Принято решение использовать:

- `ESM`;
- `NodeNext`;
- локальные импорты с `.js` в TypeScript-исходниках.

Причина:

- это убрало конфликты `CommonJS / ESM`;
- сервер стабильно запускается на `Node 22`;
- Colyseus и сопутствующие пакеты работают без `ERR_REQUIRE_ESM`.

### 7.2. Пока не писать клиент заново на C#

Принято решение:

- **не переписывать клиент сейчас на C#**;
- использовать уже установленный `godot_colyseus` и GDScript;
- сперва довести до стабильности сетевую основу.

Причина:

- задача состояла в том, чтобы получить рабочий прототип;
- GDScript-путь уже был частично готов;
- это позволило быстрее подтвердить работоспособность сервера и комнаты.

### 7.3. Пока оставить одну комнату `cats`

Принято решение:

- не дробить архитектуру на зоны, шардирование и сложные системы;
- держать один простой `Room` с `players`.

Причина:

- для раннего прототипа важнее простота и предсказуемость;
- это уменьшает число точек отказа.

## 8. Какие проблемы уже были решены

Ниже перечислены ключевые проблемы, которые уже были найдены и исправлены.

### 8.1. Конфликт ESM/CommonJS на сервере

Симптом:

- `ERR_REQUIRE_ESM`;
- ломался запуск `npm run dev`.

Решение:

- перевести сервер на единый `ESM / NodeNext` режим;
- настроить `package.json` и `tsconfig.json`;
- использовать корректные импорты.

### 8.2. Matchmaking Colyseus не работал как ожидалось

Симптом:

- Godot-клиент писал `Server unreadable`.

Причина:

- сервер поднимался не тем способом, и плагин клиента не получал ожидаемый ответ matchmaking.

Решение:

- сервер стартует через `await gameServer.listen(PORT)`;
- маршруты Colyseus matchmaking доступны корректно.

### 8.3. Плагин `godot_colyseus` ожидал старый формат ответа

Симптом:

- ошибка доступа к `response["room"]`.

Причина:

- плагин ожидал старый вложенный формат ответа;
- сервер Colyseus `0.17` отдавал плоский JSON.

Решение:

- в [client/addons/godot_colyseus/lib/client.gd](C:/Users/kuravella/Desktop/catlaw/client/addons/godot_colyseus/lib/client.gd) добавлена поддержка обоих форматов ответа.

### 8.4. Сервер падал на `@colyseus/schema`

Симптом:

- состояние комнаты не доходило до клиента;
- клиент застревал на подключении;
- внутри схемы возникали ошибки инициализации.

Причина:

- несовместимое поведение TypeScript class fields для текущей схемной модели.

Решение:

- в `tsconfig.json` включено:

```json
"useDefineForClassFields": false
```

### 8.5. Клиент пытался читать `state.players` слишком рано или не тем способом

Симптом:

- ошибки доступа к `players`;
- `players: 0` при фактическом подключении.

Решение:

- чтение карты игроков сделано через `meta_get(0)`;
- клиент синхронизирует игроков по событиям состояния и при полном обновлении.

### 8.6. Ломалось создание визуального аватара

Симптом:

- ошибка:

```text
Invalid access to property or key 'theme_override_font_sizes' on a base object of type 'Label'
```

Решение:

- в [client/scripts/cat_avatar.gd](C:/Users/kuravella/Desktop/catlaw/client/scripts/cat_avatar.gd) использован корректный API:

```gdscript
add_theme_font_size_override("font_size", 11)
```

### 8.7. После входа стрелки могли не двигать игрока

Симптом:

- пользователь подключён, но движение не ощущается.

Причина:

- фокус оставался на `LineEdit`/кнопке.

Решение:

- после входа фокус снимается с поля имени и передаётся в `WorldLayer`.

## 13. Ближайшие шаги разработки

Рекомендуемый порядок работы:

### Этап 1. Зафиксировать рабочую точку

Сделать:

- git-коммит рабочего состояния;
- короткий README с запуском;
- проверку, что проект поднимается на чистом запуске.

Цель:

- иметь точку возврата, если следующие изменения что-то сломают.

### Этап 2. Переделать движение на сервер-авторитетное

Сейчас клиент отправляет готовые координаты. Это годится только для прототипа.

Нужно сделать:

- клиент отправляет не `x/y`, а input или направление;
- сервер сам считает движение;
- сервер ограничивает скорость;
- сервер задаёт границы мира;
- клиент только отображает и сглаживает.

Это следующий наиболее важный технический шаг.

### Этап 3. Сделать минимальный мир

Нужно добавить:

- карту или фон;
- границы сцены;
- стартовые точки;
- простую камеру;
- базовую визуальную читаемость мира.

Цель:

- превратить сетевой тест в заготовку настоящей игры.

### Этап 4. Добавить реконнект

Нужно сделать:

- временное сохранение игрока после потери соединения;
- восстановление в ту же комнату;
- понятный статус `Reconnecting...`;
- возврат того же персонажа, а не создание нового.

Этим стоит заняться после стабилизации движения и базовой сцены.

### Этап 5. Ввести постоянный `playerId`

Этот этап уже закрыт в базовом виде: основной идентификатор игрока теперь `playerId`, а `sessionId` остался временной сетевой сессией.

Нужно добавить:

- постоянный `playerId`;
- гостевой аккаунт или простой логин;
- отделение пользователя от сетевой сессии.

Сейчас это реализовано как гостевой аккаунт с локально сохранённым `playerId` на клиенте и серверным профилем по `playerId`.

### Этап 6. Добавить сохранение данных

В базовом виде уже сохраняются:

- `playerId`;
- `name`;
- позиция игрока `x/y`.

Сейчас это делается через простой JSON-файл на сервере. В перспективе всё равно нужна нормальная БД.

### Этап 7. Сделать первый вертикальный срез игры

Минимальный законченный цикл, например:

- войти в мир;
- двигаться;
- видеть других игроков;
- иметь одну простую цель или взаимодействие.

Не нужно сразу делать весь MMO-комбайн.
